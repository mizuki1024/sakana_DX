# Operations Roadmap

このロードマップは、現在のPhase 1プロトタイプを、1社目PoCで使える運用レベルまで引き上げるための実行順序を定義する。

## 現在地

実装済み:

- ログイン
- ダッシュボード枠
- 水揚げ登録・一覧
- 販売登録・一覧
- 販売詳細・仕切書の簡易表示
- 仕入先・買受人の取得API

不足:

- 請求書生成
- 請求締め
- 仕入先・買受人の管理画面
- 帳票PDF/印刷品質
- 権限管理
- 監査ログ
- バックアップ・復旧手順
- 本番環境設計
- デモデータと営業資料

## Phase 0: PoC前提整理

目的:

- 1社目に提案できる業務範囲を固定する。

タスク:

- 対象顧客を「小規模漁市場・漁協・水産卸」に限定する。
- 水揚げから請求までの業務フローを1枚に整理する。
- 現場ヒアリング質問リストを作る。
- PoC価格、期間、成果条件を決める。

完了条件:

- 15分デモで説明できる業務ストーリーがある。
- PoCで作る範囲と作らない範囲が明確である。

## Phase 1: 業務MVP完成

目的:

- 紙・Excelの最低限の置き換えができる状態にする。

必須機能:

- 水揚げ登録・一覧・詳細
- 販売登録・一覧・詳細
- 仕切書の印刷
- 請求書の作成・一覧・印刷
- 仕入先管理
- 買受人管理
- CSV出力

優先実装順:

1. 買受人・仕入先のCRUD画面
2. 請求書生成API
3. 請求書一覧・詳細・印刷画面
4. 仕切書の帳票レイアウト改善
5. CSVエクスポート

完了条件:

- サンプルデータで、水揚げ登録から請求書印刷まで通せる。
- 実際の営業デモで途中停止しない。

## Phase 2: データ整合性と締め処理

目的:

- 運用中にデータが壊れない設計にする。

必須機能:

- ステータス遷移の明確化
- 請求締め
- 締め済みデータの編集制限
- 取消・訂正フロー
- 監査ログ
- 金額計算ルールの固定

設計方針:

- 売上・請求・仕切に関わるデータは原則物理削除しない。
- 締め済みデータは直接編集せず、取消または訂正で履歴を残す。
- 金額計算は再現可能にし、確定値は保存する。

完了条件:

- 二重販売、締め後変更、請求漏れを防げる。
- 操作履歴から「誰が何を変えたか」を追える。

## Phase 3: 権限・運用・セキュリティ

目的:

- 顧客データを預かれる最低限の運用体制を作る。

必須機能:

- 管理者・職員・閲覧者の権限分離
- セッション・認証設定の本番化
- DBバックアップ
- 復旧手順
- エラーログ
- 本番環境変数の管理
- 利用規約・プライバシーポリシーの草案

完了条件:

- 障害時に復旧手順を説明できる。
- 顧客に「データはどう守るのか」と聞かれて答えられる。

## Phase 4: PoC導入パッケージ

目的:

- 1社目に導入し、改善サイクルを回す。

必須成果物:

- デモ台本
- ヒアリングシート
- PoC提案書
- 初期設定手順
- サンプルデータ
- 操作マニュアル
- 問い合わせ対応フロー

完了条件:

- 初回商談から有償PoC提案まで一貫して進められる。
- 顧客の帳票・運用に合わせた追加要件を整理できる。

## Phase 5: SaaS化準備

目的:

- 2社目以降に横展開できる形にする。

必須機能:

- 組織単位のマルチテナント化
- 顧客別設定
- 帳票テンプレート設定
- 料金プラン
- 利用状況ログ
- サポート運用

完了条件:

- 個別カスタムなしで2社目に提案できる共通パッケージがある。

## 優先順位

最優先:

- 請求書生成
- 仕切書品質
- 仕入先・買受人管理
- デモデータ

次点:

- 監査ログ
- 締め処理
- CSV出力
- 権限管理

後回し:

- 決済連携
- 高度な分析
- 在庫最適化
- モバイル専用UI
- 完全なマルチテナントSaaS
