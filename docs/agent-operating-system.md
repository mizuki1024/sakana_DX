# Agent Operating System

この文書は、このプロジェクトを実運用レベルまで引き上げるためのAIエージェント運用ルールを定義する。

## 目的

「漁市場デジタル台帳」を、1社目PoCで実際に使える業務システムにする。AIエージェントは人格を増やすのではなく、専門観点ごとの成果物とレビューを出す。

## エージェント構成

### 1. Product Lead Agent

責任:

- 1社目PoCに必要な範囲を決める。
- 機能の優先順位を決める。
- 売れる状態と作り込みすぎの境界を管理する。

成果物:

- MVPスコープ
- 優先順位付きバックログ
- PoC完了条件

判断基準:

- 紙・Excel運用より明確に楽になるか。
- 1社目の導入判断に効くか。
- 今作らないと検証できないか。

### 2. Domain Analyst Agent

責任:

- 漁市場・水産卸の業務フローを整理する。
- 水揚げ、販売、仕切、請求、締め、入金、訂正の流れを仕様化する。
- 現場ヒアリングで確認すべき不明点を洗い出す。

成果物:

- 業務フロー
- 帳票要件
- 未確定ルール一覧
- ヒアリング質問リスト

重点確認:

- 手数料率
- 消費税
- 端数処理
- 締め日
- 仕切書と請求書の形式
- 訂正・取消運用

### 3. Backend and Data Agent

責任:

- データモデル、API、状態遷移、金額計算の整合性を守る。
- 請求締め後の変更、取消、監査ログを設計する。
- 本番データを壊さない実装方針を提示する。

成果物:

- DB改善案
- API仕様
- 状態遷移表
- データ移行方針

重点確認:

- 金額は再計算できるか、確定値として保存すべきか。
- 取引データを物理削除してよいか。
- 締め済みデータを編集できるか。
- 操作履歴を追えるか。

### 4. Frontend and UX Agent

責任:

- 現場入力の速さとミス防止を設計する。
- スマホ、タブレット、PCのどれを優先するか明確にする。
- 一覧、検索、印刷、登録フォームを業務順に最適化する。

成果物:

- 画面改善案
- 入力導線
- エラー表示ルール
- 印刷表示確認項目

重点確認:

- 1日の登録件数に耐える入力速度か。
- 入力ミスを戻せるか。
- 印刷前に内容確認できるか。
- 現場用語になっているか。

### 5. Security and Operations Agent

責任:

- 認証、権限、監査、バックアップ、障害対応を管理する。
- 本番公開前に最低限必要な運用設計を定義する。

成果物:

- 権限設計
- 監査ログ要件
- バックアップ・復旧手順
- 本番環境チェックリスト

重点確認:

- 管理者、職員、閲覧者の権限差。
- 個人情報・取引情報の扱い。
- DBバックアップと復旧テスト。
- エラー監視と問い合わせ対応。

### 6. QA Agent

責任:

- 実業務に基づくテストシナリオを作る。
- 正常系、異常系、境界値、帳票、印刷を検証する。

成果物:

- テストシナリオ
- リグレッション観点
- リリース判定

重点確認:

- 二重販売を防げるか。
- 締め済み請求が壊れないか。
- 手数料・税・端数が正しいか。
- 帳票が現場で使えるか。

### 7. Sales and Onboarding Agent

責任:

- 1社目PoCを取るための営業・導入資料を作る。
- 初回デモ、ヒアリング、提案、導入手順を整える。

成果物:

- デモ台本
- 初回メール
- ヒアリングシート
- PoC提案書の骨子
- 導入手順

重点確認:

- 「完成品を売る」ではなく「現場に合わせる実証導入」として提案できているか。
- 価格と範囲が明確か。
- PoC後の月額化に接続できるか。

## 実行フロー

非自明な開発タスクは以下の順で進める。

1. Product Lead Agent が目的とMVP範囲を確認する。
2. Domain Analyst Agent が業務ルールと不明点を確認する。
3. Backend and Data Agent がデータ影響を確認する。
4. Frontend and UX Agent が現場導線を確認する。
5. 実装する。
6. QA Agent が実業務シナリオで確認する。
7. Security and Operations Agent が本番運用リスクを確認する。
8. Sales and Onboarding Agent がPoC資料やデモへの影響を更新する。

## タスクの受け入れ条件

各実装タスクは、最低限以下を満たす。

- どの業務フローを改善するか明確である。
- 金額・締め・帳票・権限への影響が確認されている。
- 必要なテストまたは手動確認手順がある。
- PoCで説明できる価値がある。
- 未確定の現場ルールは仮定として文書化されている。

## 人間が判断すべきこと

AIエージェントだけで確定しない。

- 現場ごとの実際の帳票形式
- 契約金額
- 法務・税務判断
- 本番運用責任
- 顧客への正式提案
- データ移行の最終承認
